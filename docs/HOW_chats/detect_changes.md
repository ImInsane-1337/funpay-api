# Обнаружение изменений

После того как Runner выполнил инициализацию при первом запросе, он переходит в бесконечный цикл регулярных опросов серверов FunPay (с задержкой `requests_delay`).

## Механизм обнаружения изменений

При каждом новом запросе в цикле Runner вызывает метод `get_updates()` и получает JSON-структуру с изменениями с сайта FunPay.

### 1. События чатов (`type == "chat_bookmarks"`)

* Runner извлекает список чатов из полученного ответа.
* Если список чатов изменился (появился новый диалог или изменился порядок), генерируется `ChatsListChangedEvent`.
* Для каждого чата Runner сравнивает текст и время последнего полученного сообщения с данными в `Runner.last_messages`:
  * Если время или текст изменились, генерируется `LastChatMessageChangedEvent`.
  * Если `make_msg_requests` включен (по умолчанию `True`), Runner делает дополнительный запрос истории сообщений этого чата и генерирует `NewMessageEvent` для каждого нового сообщения, пришедшего с момента предыдущего опроса.

### 2. События заказов (`type == "orders_counters"`)

* Извлекаются счетчики незавершенных покупок и продаж.
* Если счетчики изменились, генерируется `OrdersListChangedEvent`.
* Если `make_order_requests` включен (по умолчанию `True`), Runner опрашивает список заказов и сравнивает состояние заказов с сохраненными в `Runner.saved_orders`:
  * Если обнаружен заказ, которого не было в кэше, генерируется `NewOrderEvent`.
  * Если статус заказа изменился (например, стал `CLOSED` вместо `PAID`), генерируется `OrderStatusChangedEvent`.

---

## Важные технические ограничения

> [!WARNING]
> Так как FunPay в списке чатов отображает только **первые 250 символов** последнего сообщения, Runner сравнивает только эту часть текста. 
> Если пользователь пришлет два сообщения подряд, у которых первые 250 символов абсолютно идентичны, и при этом время отправки (минута) совпадет, Runner может не распознать второе сообщение без полной выгрузки истории.
